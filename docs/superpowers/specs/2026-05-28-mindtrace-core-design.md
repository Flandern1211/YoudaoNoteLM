## 1. 产品概述

### 1.1 产品定位

MindTrace 是一款以"笔记本"为核心的个人知识管理与学习助手工具，对标 Google NotebookLM。用户可导入多种格式的资料，基于资料内容与 AI 对话、生成思维导图、测验、PPT 和笔记，实现从收集到内化的学习闭环。

### 1.2 目标用户

学生、研究者、知识工作者等所有需要深度处理和内化信息的人。

### 1.3 核心价值

- **来源聚合**：打破格式壁垒，将文本、网页、音频、第三方笔记等统一汇聚。
- **AI 原生处理**：不仅对话，更能将知识直接转化为导图、测验等学习资产。
- **以笔记本为单元**：按项目/主题隔离管理知识，结构清晰。

### 1.4 技术选型

| 层级 | 技术 | 说明 |
|------|------|------|
| 后端 | Go | 主要开发语言 |
| AI 框架 | Eino | Agent 编排、组件抽象（ChatModel、Retriever、Tool、Embedding 等） |
| 数据库 | PostgreSQL | 存储用户、笔记本、来源、笔记、对话等结构化数据 |
| 向量数据库 | Qdrant | 存储文档切片的向量数据 |
| Embedding | 火山引擎向量模型 | 默认 embedding 服务，通过 Eino Embedding 组件调用 |
| 语音识别 | 火山引擎 ASR | 音频转文本 |
| LLM | 用户自配置 | 支持多组 API 配置，通过 Eino ChatModel 组件调用 |
| 文档转 Markdown | MarkItDown MCP | 微软开源，通过 MCP Server（HTTP 模式）统一将 PDF/DOCX/PPTX/HTML 等格式转换为 Markdown |
| MCP 客户端 | mcp-go | 调用有道云笔记等 MCP 服务，封装为 Eino Tool |

## 2. Agent 架构设计

### 2.1 设计理念

MindTrace 采用 **"对话 Agent + 统一生成服务"** 架构。核心原则：

- **对话 Agent 是唯一的 Agent**：具备意图理解和任务调度能力，通过 ReAct 循环自主决策。
- **生成服务是共享的基础设施**：所有生成能力（思维导图、PPT、测验、笔记）封装为统一的 Generation Service，同时被 UI 按钮和对话 Agent 调用。
- **对话 Agent 不自己生成任何东西**：它识别用户意图后，调用 Generation Service 完成生成——就像"对话调用了按钮"一样。
- **双入口共享同一套生成逻辑**：UI 按钮直接调用 Generation Service，对话 Agent 也调用同一个 Generation Service，输出一致、结果统一存入笔记区域。

### 2.2 整体架构

```
┌─────────────────────┐          ┌─────────────────────┐
│    UI 工作室按钮      │          │    对话自然语言输入    │
│ （思维导图/PPT/测验/  │          │ （"帮我生成思维导图"） │
│   笔记图标）         │          │                     │
└──────────┬──────────┘          └──────────┬──────────┘
           │                                │
           │                        ┌───────┴───────┐
           │                        │   对话 Agent    │
           │                        │ 意图理解 + 调度   │
           │                        │ （ReAct 循环）   │
           │                        └───────┬───────┘
           │                                │
           │            调用同一个生成服务 ↓    │
           │                                │
    ┌──────┴────────────────────────────────┴──────┐
    │           Generation Service（生成服务）        │
    │                                              │
    │  输入：生成类型 + 选中的来源 + 参数              │
    │  执行：加载对应 skill.md → 调用 LLM → 存 Note   │
    │  输出：Note ID                                │
    │                                              │
    │  ┌─────────┐ ┌─────────┐ ┌─────────┐ ┌────┐ │
    │  │mindmap  │ │  ppt    │ │  quiz   │ │note│ │
    │  │skill.md │ │skill.md │ │skill.md │ │    │ │
    │  └─────────┘ └─────────┘ └─────────┘ └────┘ │
    └──────────────────────────┬───────────────────┘
                               │
                    ┌──────────┴──────────┐
                    │     笔记区域（右栏）    │
                    │  所有生成物统一显示在这里 │
                    └─────────────────────┘
```

### 2.3 对话 Agent

对话 Agent 是系统中唯一的 Agent，负责理解自然语言意图并将任务委托给 Generation Service。

**核心能力**：
- **意图识别**：分析用户自然语言输入，判断是普通问答、生成思维导图、出测验、生成 PPT、保存笔记，还是混合意图。
- **参数提取**：从用户消息中提取生成参数（如题目数量、PPT 风格），未指定则使用默认值。
- **工具调用**：将识别到的意图转化为对 Generation Service 的调用。
- **对话回复**：在调用生成服务后，给用户一个对话层面的反馈（如"已为你生成思维导图"）。

**Agent 工具集**：
| 工具 | 说明 |
|------|------|
| `generate` | 调用 Generation Service，参数指定生成类型（mindmap / quiz / ppt / note） |

**实现方式**：基于 Eino 的 `agent.New` 组件，配置 ChatModel + 工具列表 + 系统 Prompt。对话 Agent 通过 ReAct 循环自主决定调用哪个工具。

**关键约束**：对话 Agent 不直接执行生成逻辑，所有生成操作委托给 Generation Service。对话 Agent 的职责仅限于"理解意图 → 选择工具 → 传递参数 → 返回对话反馈"。

### 2.4 Prompt Skill

**本质**：一份精心编写的指令文档（`skill.md`），定义了 LLM 应该如何输出特定格式的内容。Skill 本身不是代码，而是**注入给 LLM 的系统 Prompt**。

**工作原理**：
```
skill.md（指令/模板/规范） + 资料内容 → 注入 LLM 系统 Prompt → LLM 按指令输出
```

**项目中的 Prompt Skill**：

| Skill | skill.md 内容 | LLM 输出格式 |
|-------|-------------|-------------|
| `mindmap` | Markmap 格式规范、层级规则、节点设计原则 | Markmap Markdown |
| `ppt` | HTML PPT 模板规范（来自 html-ppt-skill）、设计系统、布局规则、动画类 | HTML 幻灯片 |
| `quiz` | JSON 题目格式、选项设计原则、难度控制规则、输出 schema | JSON（题目数组） |
| `note` | Markdown 笔记格式规范、摘要整理原则 | Markdown |

**触发方式**：
- **UI 按钮** → 后端加载对应 skill.md + 资料内容 → 调用 LLM → 结果存 Note
- **对话 Agent** → 识别意图后调用 Generation Service → 同样的流程 → 结果存 Note

### 2.5 Generation Service（生成服务）

Generation Service 是统一的生成入口，封装了 **"模型 + skill.md"** 的调用逻辑。所有 Prompt Skill 的生成本质上都是同一件事：

```
skill.md（指令） + 资料内容 → 发给 LLM → 输出结果 → 存 Note
```

Generation Service 把这个通用流程封装一次，UI 按钮和对话 Agent 都通过它来触发生成，不需要各自重复实现。

**执行流程**：
1. 接收生成请求（类型 + 选中的来源 + 参数）
2. 根据类型加载对应的 skill.md 文件
3. 根据来源 ID 从数据库获取资料内容
4. 组装 Prompt：skill.md 作为系统 Prompt + 资料内容 + 参数
5. 调用 LLM（通过智能路由选择最优配置）
6. 将 LLM 输出保存为 Note
7. 返回生成结果（Note ID）

**请求参数**：

| 参数 | 说明 |
|------|------|
| Type | 生成类型：mindmap / quiz / ppt / note |
| SourceIDs | 选中的资料来源 ID 列表 |
| QuizCount | 测验题目数量（默认 10，仅 quiz 类型） |
| NoteMode | 笔记模式：summary / full（仅 note 类型） |

**与 UI 按钮的对应关系**：

| UI 按钮 | API 端点 | Generation Type | 加载的 skill.md |
|---------|---------|----------------|----------------|
| 🗺️ 思维导图 | `POST /api/generate` | `type=mindmap` | `skills/prompt/mindmap/skill.md` |
| 📊 生成 PPT | `POST /api/generate` | `type=ppt` | `skills/prompt/ppt/skill.md` |
| ❓ 出测验 | `POST /api/generate` | `type=quiz` | `skills/prompt/quiz/skill.md` |
| 📝 生成笔记 | `POST /api/generate` | `type=note` | `skills/prompt/note/skill.md` |

所有按钮共用同一个 API 端点，通过 `type` 参数区分生成类型。每种类型只是加载不同的 skill.md，生成流程完全一致。

**扩展新生成类型**：只需两步——① 编写 skill.md；② 在 Generation Service 中注册 type 与 skill.md 路径的映射。无需编写新的生成逻辑。

### 2.6 双入口共享机制

UI 按钮和对话触发最终调用的是同一个 Generation Service，结果都存入同一个笔记区域：

```
┌──────────────┐
│  UI 工作室    │
│  点击按钮     │ ──→ POST /api/generate ──→ Generation Service ──→ Note → 笔记区域
└──────────────┘

┌──────────────┐
│  对话输入     │
│  自然语言     │ ──→ 对话 Agent ──→ 调用 Generation Service ──→ Note → 笔记区域
└──────────────┘
```

**示例：生成思维导图**

| 入口 | 流程 |
|------|------|
| UI 按钮 | 用户选中来源 → 点击"思维导图"图标 → `POST /api/generate {type:"mindmap", source_ids:[...]}` → Generation Service 加载 mindmap skill.md → 调用 LLM → 存 Note |
| 对话触发 | 用户说"帮我把选中的内容整理成思维导图" → 对话 Agent 识别意图 → 调用 `generate_mindmap` 工具 → 同一个 Generation Service → 同样的流程 → 存 Note |

**两种入口的差异仅在于触发方式**，生成逻辑和结果存储完全相同。

### 2.7 Skill 目录结构

```
skills/

├── ppt/
│   ├── skill.md           # HTML PPT 生成指令
│   └── assets/            # PPT 模板资源（主题/布局/动画）
│       ├── base.css
│       ├── themes/

```

## 3. 导入模块

### 3.1 支持格式

| 格式 | 类型 | 大小限制 | 处理方式 |
|------|------|----------|----------|
| `.txt` | 文件 | 10MB | 直接读取文本内容 |
| `.md` | 文件 | 10MB | 直接读取，保留 Markdown 格式 |
| `.docx` | 文件 | 10MB | 结构化提取转 Markdown |
| `.pdf` | 文件 | 10MB | 结构化提取转 Markdown |
| `.mp3` | 音频 | 50MB | 火山引擎 ASR 转文本 |
| `.wav` | 音频 | 50MB | 火山引擎 ASR 转文本 |
| URL（网页） | 链接 | — | HTTP 请求 + Readability 提取正文转 Markdown |

### 3.1.1 实现方案

> **统一转换工具**：使用微软开源的 [MarkItDown](https://github.com/microsoft/markitdown) 及其 MCP Server（[markitdown-mcp](https://github.com/microsoft/markitdown)）作为所有格式 → Markdown 的转换引擎。支持 PDF、DOCX、PPTX、XLSX、HTML、图片、音频等格式，输出标准 Markdown 文本，无需为每种格式单独实现转换器。

**集成方式**：MarkItDown 自带 MCP Server，支持 STDIO / Streamable HTTP / SSE 三种传输协议。Go 后端通过 **HTTP 模式**调用：

```bash
# 启动 MCP Server（HTTP 模式）
markitdown-mcp --http --host 127.0.0.1 --port 3001
```

MCP Server 暴露工具 `convert_to_markdown(uri)`，`uri` 支持 `http:`、`https:`、`file:`、`data:` 格式。Go 端通过 HTTP 请求调用即可。

```
Go 后端 → HTTP 请求 convert_to_markdown(uri) → MCP Server → Markdown 文本 → 两阶段切片管道
```

**部署**：可使用 Docker 隔离 Python 环境依赖，Go 后端与 MCP Server 部署在同一内网中。

**注意**：MCP Server 默认绑定 localhost，不支持认证，仅适合内部可信网络使用。

**依赖**：
- 服务器安装 Python 3.10+，`pip install markitdown[mcp]`
- Go 端：标准库 `net/http` 调用 MCP Server HTTP 接口

### 3.2 解析规则

**结构化提取转 Markdown**（所有文档格式统一）：
- 保留标题层级（H1-H6）
- 保留列表（有序/无序）
- 保留表格（转为 Markdown 表格）
- 保留加粗、斜体等行内格式
- 图片：提取 alt 文本占位，不嵌入图片本身（嵌入图片链接）

### 3.2.1 实现方案

**由 MarkItDown 统一处理**：所有格式的解析和 Markdown 转换均由 MarkItDown 完成，MindTrace 不需要为每种格式单独实现转换逻辑。MarkItDown 内部已处理：
- 标题层级（H1-H6）
- 列表（有序/无序）
- 表格（转为 Markdown 表格）
- 行内格式（加粗、斜体等）
- 图片（提取 alt 文本占位）

**MindTrace 只需关注**：在 MarkItDown 输出的 Markdown 文本基础上，执行后续的**两阶段切片管道**（结构切分 → 语义精切）。

**Markdown 结构解析**：使用 `goldmark` 库解析 MarkItDown 输出的 Markdown AST，为阶段一结构切分提供支撑。

### 3.3 网址导入

- 仅支持 HTTP/HTTPS 协议
- 单页爬取，不递归爬取子页面
- 使用 Readability 算法提取正文，去除导航栏、广告等干扰内容
- 不支持需要登录才能访问的页面（V1）

### 3.3.1 实现方案

**URL 转换**：MarkItDown 内置支持 URL 输入，自动抓取网页内容并转换为 Markdown。Go 端调用方式与文件转换一致：

```bash
markitdown "https://example.com/article"
```

若 MarkItDown 对特定网页效果不佳，可回退到 `colly` 抓取 HTML + `go-readability` 提取正文 + `bluemonday` 清理 + `html-to-markdown` 转换的备用方案。

### 3.4 音频导入

- 调用火山引擎 ASR 服务进行语音识别
- 转写结果自动分段并转为 Markdown 格式
- 只支持中文

### 3.4.1 实现方案

**ASR 集成**：使用火山引擎 ASR SDK，将音频文件上传并提交转写任务，通过轮询或回调获取转写结果。

**结果处理**：将转写结果按时间戳分段，转换为 Markdown 格式，每段作为一个独立的切片。

### 3.5 有道云笔记导入

**功能描述**：
1. 在导入入口提供"绑定有道云笔记"按钮
2. 引导用户获取 API Key（网易智能开发者平台）
3. 用户粘贴 API Key 完成绑定
4. 绑定后展示其笔记目录/笔记列表
5. 支持多选笔记，一键批量导入
6. 导入后格式转为 Markdown 作为新资料来源

**备用方案**：提示用户可从有道云笔记导出 .md/.docx 文件后通过文件导入功能上传。

### 3.5.1 实现方案

**API Key 绑定流程**：
1. 用户点击"绑定有道云笔记"按钮
2. 弹窗显示引导步骤：
   - 步骤 1：点击链接打开 [网易智能开发者平台](https://mopen.163.com)
   - 步骤 2：使用手机号登录（需与有道云笔记绑定同一手机号）
   - 步骤 3：在「API 管理」中复制 API Key
   - 步骤 4：返回 MindTrace 粘贴 API Key
3. 验证 API Key 有效性后完成绑定
4. API Key 加密存储于数据库，关联到用户账号

**MCP 服务调用**：有道云笔记提供基于 MCP (Model Context Protocol) 协议的服务端点 `https://open.mail.163.com/api/ynote/mcp/sse`，通过 HTTP 请求 + `x-api-key` Header 进行认证。使用 `mcp-go` 库实现客户端，封装为 Eino Tool 组件供 Agent 调用。

**核心接口**：
- `listNotes`：获取笔记列表（支持分页、目录过滤）
- `getNoteContent`：获取指定笔记的文本内容
- `searchNotes`：搜索笔记内容

**数据同步流程**：
1. 用户选择要导入的笔记（支持多选）
2. 并发调用 `getNoteContent` 获取每篇笔记内容
3. 将返回的内容转换为 Markdown 格式
4. 创建 Source 记录，进入切片和向量化流程

**批量导入**：使用 goroutine 并发处理多篇笔记（默认并发数 5），通过 channel 收集结果，设置超时控制（默认 60 秒），避免一次性加载过多数据。

**错误处理**：
- API Key 无效：提示用户重新获取
- 网络超时：自动重试 3 次
- 笔记内容为空：跳过并记录日志

### 3.6 文档切片策略

> **核心设计**：所有格式文档先统一转为 Markdown，再经过**两阶段切片管道**——先按 Markdown 结构切分，再按语义边界精切。Markdown 的标题、列表、段落等天然提供了结构边界，是两阶段切片的基础。

#### 阶段一：按 Markdown 结构切分

以 Markdown 语法结构作为一级切分边界：

| 切分优先级 | 结构元素 | 切分规则 |
|-----------|---------|---------|
| 1 | 标题（H1-H6） | 每个标题及其下属内容作为一个切片组 |
| 2 | 列表块 | 连续的有序/无序列表作为一个独立切片 |
| 3 | 表格 | 每张 Markdown 表格作为一个独立切片 |
| 4 | 代码块 | 每个围栏代码块（` ``` `）作为一个独立切片 |
| 5 | 段落 | 连续空行分隔的段落块作为独立切片 |

**输出**：每个结构块 = 一个初级切片（struct chunk），附带层级路径（如 `# 标题A > ## 标题B`）作为上下文锚点。

#### 阶段二：按语义边界精切

对阶段一输出的每个结构切片进行二次处理：

- **过大切片**（超过阈值，默认 500 字符）：按句子边界（句号、问号、换行）二次切分，保持每段语义完整
- **过小切片**（低于阈值，默认 100 字符）：合并相邻切片，避免碎片化
- **合并后仍过大**：递归执行句子级切分

#### 切片元数据

每个最终切片附带以下上下文信息，用于增强检索和引用定位：

- `source_id`：来源文档 ID
- `chunk_index`：切片在文档中的顺序索引
- `heading_path`：所在标题层级路径（如 `# 概述 > ## 背景`）
- `file_name`：原始文件名
- `position_hint`：在原文中的大致位置（前/中/后 1/3）

### 3.6.1 实现方案

**Markdown 结构解析器**：解析 Markdown AST（Abstract Syntax Tree），按标题、列表、表格、代码块、段落等节点类型切分为结构块。使用 `goldmark` 库解析 Markdown 并遍历 AST 节点，每个顶级块节点生成一个结构切片。

**语义精切器**：实现 `SemanticSplitter` 结构体，对每个结构切片按句子边界进行二次切分，合并过小片段，递归分割过大片段。使用 Eino 的 `transformer.NewRecursiveCharacterTextSplitter` 作为基础分割工具（默认切片大小 500 字符，重叠大小 50 字符）。

**切片流水线**：定义 `ChunkPipeline` 接口，串联两个阶段：`Markdown → 结构切分 → 语义精切 → 向量化`，每个阶段的输出作为下一阶段的输入。

### 3.7 向量化存储

1. 每个切片调用火山引擎 embedding 模型生成向量
2. 向量存入 Qdrant，以 `source_id + chunk_index` 为 key
3. 原始 Markdown 内容存入 PostgreSQL
4. 向量化过程异步执行，导入后立即可用（展示加载状态）

### 3.7.1 实现方案

**向量化流程**：使用 Eino 的 Embedding 组件调用火山引擎向量模型，将文本切片转换为向量。向量数据存入 Qdrant，原始内容存入 PostgreSQL。

**异步处理**：使用 Go 的 goroutine + channel 异步执行向量化任务，通过任务状态表记录处理进度，前端轮询显示加载状态。

### 3.8 资料来源管理

**列表展示**：
- 显示：名称、类型图标（文件/URL/音频）、导入时间
- 排序：按导入时间降序

**操作**：
- **搜索**：按名称实时过滤
- **重命名**：右键/更多菜单 → 重命名
- **删除**：支持单选和多选删除，删除需二次确认
- **查看内容**：点击来源卡片，展示完整的解析后 Markdown 文本内容，显示文件名、导入时间等元数据
- **选择**：复选框多选，用于限定 AI 检索范围，底部显示已选数量

### 3.8.1 实现方案

## 4. 提问与对话模块

### 4.1 提问功能

**触发方式**：
1. 用户在左栏资料来源列表中勾选一个或多个来源
2. 在中栏对话框输入问题并发送

**处理流程**：
1. 对选中来源的内容进行向量检索（top-K 相关片段，默认 K=5）
2. 组装 Prompt：系统 prompt + 自动摘要（较早对话）+ 最近 N 轮完整对话 + 检索到的 top-K 片段 + 用户问题
3. 将 Prompt 发送给 LLM（通过智能路由选择最优配置）
4. 流式返回回答，底部显示引用的具体来源片段（来源名称 + 对应段落）

**检索策略**：
- 仅从用户当前选中的资料来源中检索
- 不跨笔记本检索
- 按相关度排序

### 4.1.1 实现方案

**向量检索**：使用 Eino 的 Retriever 组件从 Qdrant 中检索 top-K 相似片段，根据用户选中的 source_id 过滤。

**Prompt 组装**：使用 Eino 的 Chain 组件构建管道，将系统提示、对话历史摘要、检索结果、用户问题组合后发送给 LLM。

**流式输出**：使用 Eino 的 ChatModel 组件的流式接口，通过 Server-Sent Events (SSE) 向前端推送回答。

### 4.2 上下文管理

**滑动窗口 + 增量摘要混合策略**：

系统应当采用三层存储管理对话上下文：

| 层级 | 存储 | 职责 |
|------|------|------|
| 热缓存 | Redis | 缓存当前活跃对话的最近 N 轮消息和当前摘要，加速读取 |
| 持久存储 | PostgreSQL | 全量存储所有对话消息，作为唯一可信数据源 |
| 向量检索 | Qdrant | 仅用于资料来源切片的语义检索（V1 不对对话历史做向量化） |

**滑动窗口规则**：
- 最近 N 轮对话（N 由模型上下文窗口大小动态决定）保留完整内容，缓存在 Redis 中
- 窗口外的早期对话通过增量摘要压缩，摘要同样缓存在 Redis 中
- Redis 缓存未命中时，从 PostgreSQL 重建

**增量摘要策略**：
- 当有新消息滑出窗口时，将该消息追加到已有摘要中，调用 LLM 生成更新后的摘要
- 摘要设 token 上限，超过上限时触发二次压缩（将当前摘要发给 LLM 精简），防止摘要无限膨胀
- 每次生成的摘要快照持久化到 PostgreSQL，支持回溯

**Prompt 动态组装**：

Prompt 总预算应根据当前使用的模型上下文窗口大小动态分配（预留约 50% 给模型输出）：

| 组成部分 | 占 Prompt 预算比例 |
|---------|------------------|
| 系统 prompt | ~5% |
| 增量摘要 | 5% ~ 15% |
| 最近 N 轮完整对话 | 20% ~ 30% |
| 向量检索到的资料片段 | 15% ~ 25% |
| 用户当前问题 | ~5% |
| 安全余量 | 20% |

系统应当根据模型上下文窗口大小动态调整各组件的 token 预算，而非使用固定值。具体而言：
- 上下文窗口越大，保留的对话轮数 N 越多，检索的资料片段数 K 越大
- 上下文窗口越小，轮数和片段数相应缩减，摘要压缩更激进

**模型上下文窗口识别**：
- AI 配置新增 `context_window` 字段（模型上下文窗口大小，token 数）
- 新增 AI 配置时，系统根据用户填写的模型名称自动匹配内置映射表，填充上下文窗口大小
- 映射表覆盖主流模型（GPT-4o、Claude、Deepseek、Qwen 等），随版本迭代维护
- 未识别的模型名称，提示用户手动填写上下文窗口大小

**Redis 缓存淘汰**：
- 对话缓存数据设置 TTL，每次访问时刷新
- 长时间未访问的对话数据自然过期
- 同时依赖 Redis 的 LRU 淘汰策略兜底，防止内存溢出

### 4.3 对话管理

**功能列表**：
- **新建对话**：点击"新建对话"按钮，清空当前对话区域，开始新的对话
- **对话列表**：点击「☰ 对话列表」按钮，中栏左侧滑出面板，列出当前笔记本的所有历史对话（按更新时间降序）
- **切换对话**：点击对话列表中的某条对话，加载其历史消息
- **删除对话**：点击"删除"按钮，删除当前对话及其所有消息，需二次确认
- **对话标题**：默认以首条用户消息的前 20 个字符作为标题，支持手动修改

**数据存储**：
- 对话历史自动保存到数据库
- 每条消息记录：角色（user/assistant）、内容、引用来源列表、发送时间

### 4.3.1 实现方案

**CRUD 操作**：使用 GORM 或 sqlx 实现对话和消息的增删改查，对话列表按更新时间降序排列。

**标题生成**：首次对话时自动截取用户消息前 20 个字符作为标题，后续支持手动修改。

**软删除**：删除对话时使用软删除（设置 deleted_at 字段），保留历史数据可恢复。

### 4.4 保存对话为笔记

**功能描述**：
- 在对话界面提供"保存为笔记"按钮
- 点击后将当前轮次的问答内容（用户问题 + AI 回答 + 引用）打包为 Markdown 格式
- 保存为 Note（type: note），出现在笔记列表中
- 不自动转为 Source，用户可手动选择"添加到资料来源"

### 4.4.1 实现方案

**内容打包**：获取当前对话轮次的用户消息和 AI 回复，格式化为 Markdown 格式，包含引用来源信息。

**Note 创建**：创建新的 Note 记录，type 设为 note，content 存储格式化后的 Markdown 内容。

**前端更新**：保存成功后，笔记列表自动刷新显示新笔记。

## 8. 账户体系

### 8.1 注册

**字段**：邮箱、密码、邮箱验证码

**流程**：
1. 输入邮箱 → 点击"获取验证码"
2. 系统发送 6 位数字验证码到邮箱（有效期 5 分钟）
3. 填写验证码和密码（密码要求：8-20 位，至少包含字母和数字）
4. 校验通过 → 注册成功 → 跳转登录页

**约束**：
- 邮箱不可重复注册
- 验证码 60 秒内不可重复发送
- 验证码输入错误 5 次后锁定，需重新获取

### 8.1.1 实现方案

**邮箱验证**：使用 SMTP 服务发送验证码，验证码存储在 Redis 中，设置 5 分钟过期时间和 60 秒发送间隔限制。

**密码安全**：使用 bcrypt 算法对密码进行哈希存储，验证时使用 bcrypt.CompareHashAndPassword 进行比对。

### 8.2 登录

**字段**：邮箱、密码

**流程**：
1. 输入邮箱和密码
2. 弹出人机验证（滑块/点选）
3. 验证通过 → 登录成功 → 进入主页

**约束**：
- 连续输错密码 3 次，锁定账户 15 分钟
- 登录状态通过 JWT Token 维护，有效期 7 天

### 8.2.1 实现方案

**人机验证**：集成第三方验证码服务（如极验、易盾），在登录前弹出滑块或点选验证。

**JWT 认证**：使用 `golang-jwt/jwt` 库生成和验证 Token，Token 中包含用户 ID 和过期时间，存储在客户端 localStorage。

**登录限制**：使用 Redis 记录登录失败次数，超过 3 次自动锁定 15 分钟。

### 8.3 找回密码

**字段**：邮箱、邮箱验证码、新密码

**流程**：
1. 输入邮箱 → 获取验证码
2. 填写验证码和新密码
3. 校验通过 → 密码重置成功 → 跳转登录页

### 8.3.1 实现方案

**验证码验证**：复用注册流程的验证码逻辑，从 Redis 中获取并验证验证码的有效性和错误次数。

**密码重置**：验证通过后，使用 bcrypt 哈希新密码并更新到数据库，同时清除该用户的所有登录会话。

## 9. 设置模块

### 9.1 AI 服务配置

**功能**：
- 可添加多组 AI 配置，每组包含：API_KEY、Model、URL
- 每组配置可独立启用/禁用
- 配置保存后立即生效，无需重启

### 9.1.1 实现方案

**配置存储**：AIConfig 实体存储在 PostgreSQL 中，支持 CRUD 操作。

**热更新**：配置变更后通过内存缓存（如 sync.Map）立即生效，无需重启服务。每次请求时根据配置动态创建 Eino ChatModel 实例。

**配置校验**：保存前通过 Eino ChatModel 发送测试请求验证 API_KEY 有效性。

### 9.2 智能路由

**策略**：
- 系统每 5 分钟探测各启用配置的健康状态和响应时间
- 探测方式：向各 URL 发送轻量级请求（如 GET /models）
- 探测结果缓存，请求时直接从缓存选最优配置
- 选择策略：健康且响应时间最短的配置
- 某配置连续 3 次探测失败，标记为不健康，自动切换到备用配置
- 配置恢复健康后自动重新加入路由池

### 9.2.1 实现方案

**健康探测**：使用 Go 的 `net/http` 客户端发送轻量级请求，记录响应时间和状态，结果缓存在 Redis 中。

**路由选择**：实现 `Router` 结构体，维护健康配置列表，每次请求时从中选择响应时间最短的配置。使用 mutex 保证并发安全。

**自动恢复**：探测 goroutine 定期检查不健康配置，恢复后自动重新加入路由池。

## 10. 个人中心

### 10.1 头像

- 支持上传自定义头像（jpg/png，最大 2MB）
- 提供裁剪功能
- 未上传时使用默认头像

### 10.1.1 实现方案

**文件上传**：使用 Go 的 `multipart` 处理文件上传，限制文件类型和大小，存储到本地或 OSS。

**图片裁剪**：前端使用 `react-image-crop` 或 `vue-cropper` 实现裁剪功能，裁剪后上传到服务器。

**头像存储**：将裁剪后的图片保存到对象存储（如阿里云 OSS），数据库存储 URL 引用。

### 10.2 邮箱

- 仅作展示，不可修改

### 10.3 修改密码

- 需验证当前密码
- 输入新密码并确认
- 修改成功后需重新登录

### 10.3.1 实现方案

**密码验证**：使用 bcrypt 验证当前密码是否正确，验证通过后才允许修改。

**更新密码**：对新密码进行 bcrypt 哈希处理，更新到数据库，同时使当前 JWT Token 失效。

**重新登录**：修改成功后返回特定状态码，前端清除 Token 并跳转到登录页。

## 11. 非功能性需求

### 11.1 性能

- 文件导入：10MB 文档解析 + 向量化应在 30 秒内完成
- 提问响应：流式输出，首 token 响应时间 < 3 秒
- 页面加载：主页 < 2 秒，笔记本详情页 < 3 秒

### 11.2 安全

- 密码使用 bcrypt 哈希存储
- API 传输使用 HTTPS
- JWT Token 包含用户 ID，服务端校验
- 文件上传校验格式和大小，防止恶意文件

### 11.3 可用性

- 支持 Chrome、Firefox、Safari 最新两个主要版本
- 响应式布局，最小支持 1280px 宽度
- 深色主题（默认），可考虑后续支持浅色主题
